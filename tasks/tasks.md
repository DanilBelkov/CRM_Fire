# Задачи по доработке интерфейса и логики приложения

## 1. Окно редактирования

### 1.1. Галерея сущности

- [ ] При добавлении фото в галерею сущности отображать добавленные изображения в поле `ListBox`.
- [ ] Для каждого изображения в `ListBox` показывать понятное представление:
  - миниатюра изображения;
  - название файла или краткое описание;
  - возможность выбрать изображение из списка.
- [ ] После добавления нового фото сразу обновлять список изображений без перезапуска окна.
- [ ] При удалении фото из галереи удалять его также из `ListBox`.

**Критерии приемки:**

- Добавленное изображение сразу появляется в `ListBox`.
- При повторном открытии сущности список фотографий отображается корректно.
- Если у сущности нет фото, список отображается пустым или с понятной заглушкой.

---

### 1.2. Поле «Ответственный»

- [ ] В `ComboBox` поля «Ответственный» добавить возможность удалить выбранного пользователя.
- [ ] Реализовать удаление через кнопку на item внутри списка `ComboBox`.
- [ ] При нажатии на кнопку удаления пользователь удаляется из списка ответственных.
- [ ] После удаления обновлять значение поля, если удаленный пользователь был выбран текущим ответственным.

**Критерии приемки:**

- У каждого пользователя в выпадающем списке есть кнопка удаления.
- Удаление пользователя не ломает выбор других пользователей.
- Если удаляется текущий выбранный пользователь, поле очищается или выбирается значение по умолчанию.

---

## 2. Главное окно

### 2.1. Раздел настроек

- [ ] Объединить функции:
  - экспорт Excel;
  - импорт Excel;
  - восстановление backup.
- [ ] Обернуть эти функции в один общий блок.
- [ ] Сделать этот блок сворачиваемым разделом настроек.
- [ ] По умолчанию раздел настроек может быть свернут.

**Критерии приемки:**

- Экспорт, импорт и восстановление backup находятся в одном визуальном блоке.
- Блок можно свернуть и раскрыть.
- Интерфейс главного окна стал менее перегруженным.

---

### 2.2. Удаление создания вкладок

- [ ] Удалить функциональность создания вкладок из главного окна.
- [ ] Удалить связанные кнопки, обработчики и неиспользуемую логику.
- [ ] Проверить, что удаление создания вкладок не ломает работу существующих разделов.

**Критерии приемки:**

- Пользователь больше не может создавать вкладки.
- В интерфейсе нет кнопок или элементов, связанных с созданием вкладок.
- Существующие разделы продолжают открываться корректно.

---

### 2.3. Превью карточки

- [ ] В `DetailsImage` отображать первое фото из списка галереи сущности.
- [ ] Если фото в галерее нет, отображать заглушку или пустое состояние.
- [ ] При изменении галереи обновлять превью карточки.

**Критерии приемки:**

- В превью карточки отображается первое изображение из галереи.
- Если первое изображение удалили, превью обновляется на следующее доступное.
- Если изображений нет, интерфейс не падает.

---

### 2.4. Кнопки «Удалить навсегда» и «Восстановить»

- [ ] Показывать кнопки «Удалить навсегда» и «Восстановить» только в разделе «Корзина».
- [ ] В остальных разделах скрыть эти кнопки.
- [ ] Проверить, что действия доступны только для элементов, находящихся в корзине.

**Критерии приемки:**

- В обычных разделах кнопки не отображаются.
- В разделе «Корзина» обе кнопки доступны.
- Нельзя случайно удалить навсегда или восстановить элемент вне корзины.

---

### 2.5. Шторка между списком и превью карточки

- [ ] Добавить возможность двигать разделитель между списком и превью карточки.
- [ ] Ограничить минимальную и максимальную ширину областей.
- [ ] Предотвратить ситуацию, когда список или превью полностью исчезают.
- [ ] Сохранять удобное поведение при изменении размера окна.

**Критерии приемки:**

- Пользователь может менять ширину списка и превью.
- Области имеют минимальные размеры.
- При изменении размера окна интерфейс остается usable.

---

### 2.6. История

- [ ] Свернуть блок истории по умолчанию.
- [ ] Добавить возможность раскрывать и сворачивать историю.
- [ ] Добавить вертикальный скролл для длинной истории.
- [ ] Добавить горизонтальный скролл для широкого содержимого истории.
- [ ] Ограничить высоту раскрытого блока истории, чтобы он не занимал все окно.

**Критерии приемки:**

- История не перегружает интерфейс при открытии карточки.
- Пользователь может раскрыть историю при необходимости.
- Длинные и широкие записи истории можно просматривать через скролл.

## 3. Окно фильтра

### 3.1. Замена создания вкладок на окно фильтра

- [ ] Вместо создания вкладок реализовать окно фильтра для выбранного раздела.
- [ ] Окно фильтра должно открываться относительно текущего выбранного раздела.
- [ ] Поля фильтра должны формироваться по полям сущности выбранного раздела.
- [ ] После применения фильтра список сущностей должен обновляться.
- [ ] Добавить возможность сбросить фильтр.

**Критерии приемки:**

- Пользователь выбирает раздел и открывает фильтр именно для этого раздела.
- Фильтр содержит поля, соответствующие сущности выбранного раздела.
- Поиск выполняется по значениям полей сущности.

---

### 3.2. Жизненный цикл фильтра

- [ ] Фильтр не должен сохраняться для дальнейшего использования.
- [ ] Фильтр живет только пока запущено приложение.
- [ ] После закрытия и повторного запуска приложения фильтр должен быть сброшен.
- [ ] Не сохранять фильтр в базу данных, настройки или файл конфигурации.

**Критерии приемки:**

- После перезапуска приложения фильтры не восстановлены.
- Во время текущей сессии приложения фильтр может оставаться активным до сброса.
- Сохраненных пользовательских фильтров нет.

---

## 4. Поля сущностей

### 4.1. Поле «Передвижение предмета»

- [ ] Для сущности «Одежда» добавить поле «Передвижение предмета».
- [ ] Для сущности «Реквизит» добавить поле «Передвижение предмета».
- [ ] Для сущности «Костюм» добавить поле «Передвижение предмета».
- [ ] В поле должно описываться:
  - кто взял предмет;
  - куда взял предмет;
  - на какой срок взял предмет.
- [ ] Добавить отображение поля в карточке сущности.
- [ ] Добавить редактирование поля в окне редактирования.
- [ ] Добавить поддержку поля в фильтре выбранного раздела.

**Критерии приемки:**

- Поле есть у одежды, реквизита и костюмов.
- Значение поля можно создать, изменить и просмотреть.
- По этому полю можно искать через фильтр.

---

### 4.2. Поле «Габариты» для реквизита

- [ ] Для сущности «Реквизит» изменить поле «Габариты» на select.
- [ ] Добавить значения:
  - маленький;
  - средний;
  - большой.
- [ ] Определить размеры категорий от маленького предмета, например кольца, до крупного предмета, например разобранной палатки.
- [ ] Обновить форму создания и редактирования реквизита.
- [ ] Обновить отображение карточки реквизита.
- [ ] Обновить фильтр по реквизиту с учетом select-поля.

**Предварительная логика категорий:**

- `Маленький` — помещается в ладонь или небольшую коробку, например кольцо, украшение, небольшой аксессуар.
- `Средний` — переносится одним человеком без разборки, например шляпа, сумка, небольшой ящик, элемент декора.
- `Большой` — требует много места, переноски двумя людьми или разборки, например крупная декорация, чемодан, разобранная палатка.

**Критерии приемки:**

- Пользователь выбирает габарит из списка, а не вводит вручную.
- Старые значения поля корректно обработаны или приведены к новой структуре.
- Фильтр может искать реквизит по габаритам.

---

### 4.3. Поле «Хрупкость» для реквизита

- [ ] Для сущности «Реквизит» изменить поле «Хрупкость» на select.
- [ ] Добавить степени хрупкости.
- [ ] Обновить форму создания и редактирования реквизита.
- [ ] Обновить отображение карточки реквизита.
- [ ] Обновить фильтр по реквизиту с учетом select-поля.

**Предварительная шкала хрупкости:**

- `Низкая` — предмет прочный, не требует особой упаковки.
- `Средняя` — предмет может повредиться при падении или сильном давлении.
- `Высокая` — предмет требует аккуратной транспортировки и отдельной упаковки.
- `Очень высокая` — предмет крайне хрупкий, желательно перевозить отдельно и с маркировкой.

**Критерии приемки:**

- Пользователь выбирает хрупкость из списка.
- Значения хрупкости единообразны.
- Фильтр может искать реквизит по степени хрупкости.

## 6. Уточняющие вопросы

1. В поле «Ответственный» нужно удалять пользователя только из текущей сущности или полностью из общего справочника пользователей? - нет, удалить только с сущности.
2. Если удаляется ответственный, который уже указан в других сущностях, нужно ли очищать его там тоже? - нет
3. В `ListBox` галереи нужно показывать только миниатюры или миниатюру вместе с названием файла? - показывать только миниатюры
4. Нужно ли менять порядок фото в галерее вручную, чтобы пользователь мог выбрать, какое фото будет первым и попадет в `DetailsImage`? - Добавить возможность удалить фото 
5. Для блока настроек в главном окне: он должен быть свернут по умолчанию всегда или запоминать последнее состояние в рамках текущего запуска приложения? - свернут всегда
6. Фильтр при смене раздела должен сбрасываться автоматически или храниться отдельно для каждого раздела до закрытия приложения? - фильтры хранятся отдельно до закрытия приложения 
7. Поле «Передвижение предмета» должно быть одним текстовым полем или структурой из отдельных полей: «кто», «куда», «дата выдачи», «дата возврата»? - сделать одним текстовым полем с текстом подсказкой «кто», «куда», «дата выдачи», «дата возврата»
8. Нужно ли вести историю передвижения предмета или хранить только текущее передвижение? - только текущее передвижение. История будет отображаться в глобальной истории
9. Для поля «Габариты» у реквизита достаточно трех значений: маленький, средний, большой, или нужно добавить, например, «очень большой»? - достаточно трех
10. Для поля «Хрупкость» подходит шкала `Низкая / Средняя / Высокая / Очень высокая`, или нужны другие названия? - этого хватит 
11. Нужно ли мигрировать уже существующие данные в полях «Габариты» и «Хрупкость» к новым значениям select? - нет, сделай их по дефолту первыми
12. Какой целевой размер шрифта использовать: например 14, 16 или 18 px? - 14 хватит
13. Нужно ли увеличивать шрифт только в интерфейсе или также в экспортируемых Excel-файлах? - да, сделай единым все
