# PRD: Desktop CRM для театральных вещей и реквизита

## 1. Introduction / Overview

Нужно создать переносимое desktop-приложение для учета театральных вещей: одежды, реквизита, костюмов, спектаклей и тегов. Приложение должно работать локально с флешки или внешнего диска, хранить базу и фотографии рядом с исполняемым файлом, не требовать сервера, интернета, авторизации или администратора.

Основная задача продукта - дать сотрудникам костюмерной и реквизиторской удобный инструмент для быстрого поиска, группировки, связывания и экспорта информации о вещах. Теги являются главным механизмом семантической организации данных: через них пользователь настраивает группы, фильтры, вкладки, панели и представления под свои рабочие процессы.

Ориентир по гибкости тегов и фильтрации - YouTrack: пользователь сам заводит смысловые группы тегов, назначает теги сущностям и строит представления на их основе.

## 2. Goals

- Создать локальное переносимое CRM-приложение для Windows на простом технологическом стеке: .NET, C#, WPF, SQLite.
- Обеспечить CRUD для одежды, реквизита, костюмов, спектаклей и тегов.
- Поддержать связи многие-ко-многим между одеждой, реквизитом, костюмами и спектаклями.
- Сделать теги центральным инструментом группировки, фильтрации и пользовательской настройки интерфейса.
- Реализовать быстрый поиск по названию и описанию внутри каждого раздела.
- Реализовать экспорт всей базы в Excel с фотографиями и без фотографий.
- Обеспечить переносимость приложения: все данные, настройки и изображения хранятся рядом с приложением на флешке.
- Сделать современный минималистичный темный интерфейс с зеленым акцентом.
- Обеспечить стабильную работу для сценария "несколько сотрудников используют одну флешку по очереди".

## 3. Target Users

- Сотрудник костюмерной.
- Реквизитор.
- Помощник режиссера или администратор постановок, которому нужно быстро понимать состав спектакля.
- Сотрудник театра, который временно работает с переносной базой без технических навыков.

## 4. Product Scope

### In Scope

- Локальное desktop-приложение для Windows.
- Работа без интернета и без сервера.
- Хранение данных в SQLite.
- Хранение фото в файловой структуре рядом с базой.
- CRUD для всех основных сущностей.
- Связи между сущностями.
- Теги, группы тегов и безопасное изменение/удаление тегов.
- Пользовательские вкладки, фильтры и панели на основе тегов.
- Поиск по названию и описанию.
- Группировка списков по группам тегов и конкретным тегам.
- Экспорт в Excel.
- Резервное копирование.
- История изменений.
- Импорт из Excel/CSV как расширенная функция полной версии.

### Out of Scope

- Многопользовательская одновременная работа по сети.
- Серверная синхронизация.
- Облачное хранилище.
- Авторизация, роли, администраторы и права доступа.
- Мобильные приложения.
- Web-интерфейс.
- Интеграция с билетными системами, бухгалтерией или складскими системами.

## 5. Core Concepts

### 5.1 Базовая сущность

Для упрощения модели все основные объекты должны наследовать общий набор полей от базовой сущности `CatalogItem`.

Общие поля:

- `Id` - уникальный идентификатор.
- `Type` - тип сущности: `Clothing`, `Prop`, `Costume`, `Performance`.
- `Title` - название.
- `Description` - описание.
- `MainPhotoId` - основная фотография.
- `InventoryNumber` - инвентарный номер, опционально.
- `StorageLocation` - физическое место хранения, опционально.
- `Condition` - состояние: новое, хорошее, требует ремонта, списано, другое.
- `ResponsiblePerson` - ответственный сотрудник, опционально.
- `Status` - активный, архивный, списанный.
- `CreatedAt` - дата создания.
- `UpdatedAt` - дата последнего изменения.
- `DeletedAt` - дата мягкого удаления, если объект помещен в корзину.
- `Notes` - служебные заметки.

Отдельные сущности могут добавлять свои специфичные поля.

### 5.2 Сущности

#### Одежда

Поля:

- Общие поля `CatalogItem`.
- `Size` - размер, опционально.
- `GenderPresentation` - мужское, женское, унисекс, детское, другое.
- `Color` - основной цвет, опционально.
- `Material` - материал, опционально.
- `Measurements` - мерки или размеры в свободном формате.
- `CareInstructions` - рекомендации по уходу.
- `Tags` - список тегов.
- `Photos` - список фото.
- `LinkedCostumes` - костюмы, где используется одежда.
- `LinkedPerformancesDirectly` - спектакли, куда одежда добавлена напрямую без костюма.

#### Реквизит

Поля:

- Общие поля `CatalogItem`.
- `Dimensions` - габариты, опционально.
- `Weight` - вес, опционально.
- `Material` - материал, опционально.
- `Fragility` - хрупкость: обычный, хрупкий, особо хрупкий.
- `UsageNotes` - особенности использования.
- `Tags` - список тегов.
- `Photos` - список фото.
- `LinkedCostumes` - костюмы, где используется реквизит.
- `LinkedPerformancesDirectly` - спектакли, куда реквизит добавлен напрямую.

#### Костюм

Поля:

- Общие поля `CatalogItem`.
- `CharacterName` - персонаж, опционально.
- `ActorName` - актер, опционально.
- `SceneNotes` - сцены или моменты использования.
- `ClothingItems` - список элементов одежды.
- `Props` - список реквизита.
- `Tags` - список тегов.
- `Photos` - список фото.
- `LinkedPerformances` - спектакли, где используется костюм.

#### Спектакль

Поля:

- Общие поля `CatalogItem`.
- `Director` - режиссер, опционально.
- `PremiereDate` - дата премьеры, опционально.
- `Season` - сезон, опционально.
- `Costumes` - список костюмов.
- `PropsDirectly` - реквизит, добавленный напрямую в спектакль.
- `ClothingDirectly` - одежда, добавленная напрямую в спектакль без костюма.
- `Tags` - список тегов.
- `Photos` - список фото.

#### Теги

Поля:

- `Id` - уникальный идентификатор.
- `Name` - название.
- `Description` - описание.
- `GroupId` - группа тега.
- `Color` - цвет тега.
- `Icon` - иконка тега, опционально.
- `SortOrder` - порядок отображения.
- `IsArchived` - архивирован ли тег.
- `CreatedAt` - дата создания.
- `UpdatedAt` - дата последнего изменения.

#### Группа тегов

Поля:

- `Id` - уникальный идентификатор.
- `Name` - название группы, например "Сектор хранения", "Эпоха", "Сезон", "Состояние", "Материал".
- `Description` - описание.
- `Color` - цвет группы.
- `SortOrder` - порядок отображения.
- `IsRequired` - обязательна ли группа для выбранных типов сущностей.
- `AllowedEntityTypes` - типы сущностей, где применима группа.
- `CreatedAt`.
- `UpdatedAt`.

#### Пользовательское представление

Пользовательские представления нужны для настройки вкладок, фильтров и панелей без администрирования.

Поля:

- `Id`.
- `Name`.
- `SectionType` - раздел: одежда, реквизит, костюмы, спектакли.
- `ViewType` - список, карточки, таблица.
- `PinnedTagGroupIds` - закрепленные группы тегов.
- `DefaultTagFilters` - фильтры по умолчанию.
- `VisibleFields` - отображаемые поля.
- `SortMode` - сортировка.
- `GroupByTagGroupId` - группировка по группе тегов.
- `IsShownInSidebar` - показывать как отдельную вкладку в левой панели.
- `SortOrder`.

## 6. Relationships

- Одежда может входить в несколько костюмов.
- Одежда может быть добавлена напрямую в несколько спектаклей без костюма.
- Реквизит может входить в несколько костюмов.
- Реквизит может быть добавлен напрямую в несколько спектаклей.
- Костюм может входить в несколько спектаклей.
- Спектакль содержит много костюмов.
- Спектакль содержит много реквизита напрямую.
- Спектакль может содержать много одежды напрямую.
- Любая основная сущность может иметь много тегов.
- Один тег может быть назначен многим сущностям.
- Один тег относится к одной группе тегов.
- Группа тегов содержит много тегов.

## 7. User Stories

### US-001: Создание базовой структуры локального приложения
**Description:** As a user, I want приложение запускалось с флешки и сохраняло данные рядом с собой so that я мог переносить его между компьютерами.

**Acceptance Criteria:**
- [ ] Приложение запускается как Windows desktop app без сервера.
- [ ] SQLite база создается рядом с приложением в папке `data`.
- [ ] Фотографии сохраняются в папке `data/photos`.
- [ ] Настройки интерфейса сохраняются в папке `data/settings` или в SQLite.
- [ ] При переносе всей папки приложения на другой компьютер данные остаются доступны.
- [ ] При отсутствии базы приложение создает пустую базу автоматически.

### US-002: CRUD для одежды
**Description:** As a костюмер, I want создавать и редактировать карточки одежды so that вести полный учет вещей.

**Acceptance Criteria:**
- [ ] Пользователь может создать карточку одежды.
- [ ] Пользователь может заполнить название, описание, фото, теги, место хранения, состояние, инвентарный номер и ответственного.
- [ ] Пользователь может редактировать карточку.
- [ ] Пользователь может отправить карточку в корзину.
- [ ] Карточка показывает связанные костюмы и спектакли как кликабельные ссылки.
- [ ] Поиск в разделе одежды ищет по названию и описанию.

### US-003: CRUD для реквизита
**Description:** As a реквизитор, I want создавать карточки реквизита so that быстро находить предметы и понимать, где они используются.

**Acceptance Criteria:**
- [ ] Пользователь может создать карточку реквизита.
- [ ] Пользователь может заполнить название, описание, фото, теги, место хранения, состояние, инвентарный номер и ответственного.
- [ ] Пользователь может указать габариты, вес, материал и хрупкость.
- [ ] Карточка показывает связанные костюмы и спектакли как кликабельные ссылки.
- [ ] Поиск в разделе реквизита ищет по названию и описанию.

### US-004: CRUD для костюмов
**Description:** As a костюмер, I want собирать костюм из одежды и реквизита so that видеть полный состав образа.

**Acceptance Criteria:**
- [ ] Пользователь может создать карточку костюма.
- [ ] Пользователь может добавить в костюм несколько элементов одежды.
- [ ] Пользователь может добавить в костюм несколько элементов реквизита.
- [ ] Один элемент одежды может быть добавлен в несколько костюмов.
- [ ] Один реквизит может быть добавлен в несколько костюмов.
- [ ] Карточка костюма показывает все связанные спектакли как кликабельные ссылки.
- [ ] Поиск в разделе костюмов ищет по названию и описанию.

### US-005: CRUD для спектаклей
**Description:** As a сотрудник театра, I want формировать состав спектакля so that видеть все костюмы, вещи и реквизит постановки.

**Acceptance Criteria:**
- [ ] Пользователь может создать карточку спектакля.
- [ ] Пользователь может добавить в спектакль несколько костюмов.
- [ ] Один костюм может быть добавлен в несколько спектаклей.
- [ ] Пользователь может добавить реквизит напрямую в спектакль.
- [ ] Пользователь может добавить одежду напрямую в спектакль без костюма.
- [ ] Карточка спектакля показывает состав с разделением на костюмы, прямой реквизит и прямую одежду.
- [ ] Поиск в разделе спектаклей ищет по названию и описанию.

### US-006: Управление тегами и группами тегов
**Description:** As a user, I want создавать группы тегов и теги so that организовывать базу под реальные процессы театра.

**Acceptance Criteria:**
- [ ] Пользователь может создать группу тегов.
- [ ] Пользователь может создать тег внутри группы.
- [ ] Пользователь может назначить цвет тега.
- [ ] Пользователь может назначить тег любой основной сущности.
- [ ] Пользователь может редактировать название, описание, цвет и группу тега.
- [ ] При переименовании тега все связи с сущностями сохраняются.
- [ ] При перемещении тега в другую группу все связи с сущностями сохраняются.

### US-007: Безопасное удаление и архивирование тегов
**Description:** As a user, I want удалять или архивировать теги без потери карточек so that изменение классификации не ломало базу.

**Acceptance Criteria:**
- [ ] При удалении тега приложение показывает количество связанных сущностей.
- [ ] Пользователь может выбрать действие: архивировать тег, заменить другим тегом, удалить тег из сущностей.
- [ ] Удаление тега не удаляет одежду, реквизит, костюмы или спектакли.
- [ ] Архивный тег не предлагается по умолчанию при назначении, но остается видимым в старых карточках.
- [ ] История изменений фиксирует удаление, замену или архивирование тега.

### US-008: Настройка вкладок, фильтров и панелей
**Description:** As a user, I want настроить рабочее пространство под себя so that быстро открывать нужные срезы базы.

**Acceptance Criteria:**
- [ ] Пользователь может создать пользовательское представление для раздела.
- [ ] Представление может отображаться как отдельная вкладка в левой панели.
- [ ] Представление может иметь фильтры по тегам по умолчанию.
- [ ] Представление может группировать список по выбранной группе тегов.
- [ ] Пользователь может выбрать вид: список, карточки или таблица.
- [ ] Пользователь может выбрать видимые поля.
- [ ] Настройки сохраняются локально и остаются после перезапуска.
- [ ] Для настройки не нужны роли, администратор или отдельный режим управления.

### US-009: Группировка по тегам
**Description:** As a user, I want группировать список по группе тегов so that видеть вещи по секторам, сезонам, эпохам или другим смысловым категориям.

**Acceptance Criteria:**
- [ ] В каждом разделе можно выбрать группу тегов для группировки.
- [ ] Если выбрана группа тегов, список разбивается на группы по тегам этой группы.
- [ ] Если выбрана группа тегов и конкретный тег, отображаются только сущности с этим тегом.
- [ ] Сущности без тега из выбранной группы попадают в группу "Без тега".
- [ ] Поиск работает внутри текущего раздела и учитывает активные фильтры.

### US-010: Поиск по названию и описанию
**Description:** As a user, I want искать по названию и описанию so that быстро находить нужные карточки.

**Acceptance Criteria:**
- [ ] В каждом разделе есть строка поиска.
- [ ] Поиск выполняется только в активном разделе.
- [ ] Поиск ищет по названию и описанию.
- [ ] Поиск работает вместе с фильтрами тегов.
- [ ] Результаты обновляются без перезагрузки приложения.
- [ ] При пустом результате показывается понятное пустое состояние.
- [ ] Для SQLite используется FTS5 полнотекстовый индекс.

### US-011: Расширенный семантический поиск
**Description:** As a user, I want находить похожие по смыслу описания so that не зависеть только от точных слов.

**Acceptance Criteria:**
- [ ] Полнотекстовый поиск FTS5 является базовым обязательным поиском.
- [ ] Семантический поиск реализуется как опциональная функция полной версии.
- [ ] Семантический поиск работает локально без внешнего API.
- [ ] Если локальная модель недоступна, приложение продолжает использовать FTS5.
- [ ] Пользователь явно видит режим поиска: обычный или смысловой.

### US-012: Фото и галерея
**Description:** As a user, I want добавлять фотографии к карточкам so that визуально идентифицировать вещи.

**Acceptance Criteria:**
- [ ] Пользователь может добавить одну или несколько фотографий к карточке.
- [ ] Первая фотография используется как обложка.
- [ ] Пользователь может заменить обложку.
- [ ] Пользователь может удалить фото из карточки.
- [ ] Фото физически хранится в `data/photos`.
- [ ] База хранит относительный путь к фото, а не абсолютный путь.
- [ ] При переносе папки приложения фотографии остаются доступны.

### US-013: Экспорт в Excel
**Description:** As a user, I want выгрузить базу в Excel so that передавать данные коллегам или делать отчетность.

**Acceptance Criteria:**
- [ ] Пользователь может экспортировать всю базу.
- [ ] Пользователь может выбрать экспорт с фотографиями или без фотографий.
- [ ] Экспорт без фотографий создает компактный `.xlsx`.
- [ ] Экспорт с фотографиями добавляет изображения или ссылки на изображения по выбранному режиму.
- [ ] В Excel есть отдельные листы: одежда, реквизит, костюмы, спектакли, теги, связи.
- [ ] В экспорт попадают связи между сущностями.
- [ ] Пользователь видит результат экспорта и путь к файлу.

### US-014: Импорт из Excel/CSV
**Description:** As a user, I want импортировать существующие списки so that быстро перенести старый учет в приложение.

**Acceptance Criteria:**
- [ ] Пользователь может импортировать одежду и реквизит из `.xlsx` или `.csv`.
- [ ] Перед импортом показывается предпросмотр.
- [ ] Пользователь может сопоставить колонки файла с полями приложения.
- [ ] Приложение показывает ошибки строк до записи в базу.
- [ ] Импорт не создает дубликаты при совпадении инвентарного номера.

### US-015: История изменений
**Description:** As a user, I want видеть историю изменений карточек so that понимать, кто и когда менял важные данные.

**Acceptance Criteria:**
- [ ] Для каждой карточки хранится история изменений.
- [ ] История показывает дату, тип изменения и измененные поля.

## 10. Technical Requirements

### Recommended Stack

- Platform: Windows desktop.
- Language: C#.
- UI: WPF.
- Runtime: .NET 8 LTS или более новая LTS-версия .NET.
- Database: SQLite.

### Database Considerations

- Все связи многие-ко-многим должны храниться через отдельные join-таблицы.

- Для поиска создать FTS5-таблицы или виртуальные индексы по `Title` и `Description`.

- Для пользовательских представлений использовать отдельную таблицу `SavedViews`.

## 12. Search Strategy

Базовый поиск должен быть построен на SQLite FTS5, потому что это простой, быстрый и локальный механизм, который хорошо подходит под стек SQLite/.NET.

Векторный или семантический поиск стоит рассматривать как дополнительную функцию полной версии, а не как основу первой реализации. Причина: локальный векторный поиск потребует модели эмбеддингов, индекса, дополнительной упаковки и усложнит переносимость. Для пользовательского запроса "по названию и вхождению в текст описания" FTS5 достаточно и надежнее.

Рекомендуемый подход:

- MVP/full base: SQLite FTS5 по `Title`, `Description`, `InventoryNumber`, `Notes`.
- Расширение: локальные эмбеддинги через ONNX-модель и хранение векторов в SQLite или отдельном локальном индексе.
- UI должен явно разделять обычный поиск и смысловой поиск, если семантический режим будет добавлен.

- Автоматическое резервное копирование должно быть настраиваемым.

- Все удаления должны быть мягкими, кроме явного окончательного удаления из корзины.

## 17. Risks and Mitigations

- Risk: Фотографии потеряются при переносе.
  Mitigation: Использовать только относительные пути и хранить фото внутри `data/photos`.

- Risk: Пользователь удалит важный тег и потеряет классификацию.
  Mitigation: Архивирование по умолчанию, подтверждение, замена тегов, история изменений.

- Risk: Векторный поиск усложнит установку и переносимость.
  Mitigation: Сделать FTS5 обязательной основой, семантический поиск - опциональным расширением.

- Risk: Большие списки будут тормозить в WPF.
  Mitigation: Использовать UI virtualization, пагинацию или incremental loading.

- Risk: Флешка может быть повреждена.
  Mitigation: Простые резервные копии в архив и понятная функция восстановления.
